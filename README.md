
## Цей репозиторій містить рішення завдань із системного програмування під Linux мовою C (Практична робота №8). Робота присвячена поглибленому вивченню низькорівневих системних викликів для роботи з файлами (`write`, `read`, `lseek`), управління процесами (`fork`) та дослідження поведінки системних функцій (`qsort`).

## 🚀 Індивідуальне завдання (Варіант 11)
Task 8.11 (task8_var11.c) — Аналіз часу виконання qsort() та пояснення сплесків:Завдання: Побудувати таблицю з часами виконання qsort() для різних схем вхідних даних (5 млн елементів) і пояснити сплески в часі.Результати та АналізПрограма генерує масиви для 6 різних конфігурацій та вимірює час сортування у мілісекундах.Природа "сплесків" у часі виконання:Теоретична вразливість Quick Sort: У наївній реалізації, якщо опорний елемент (pivot) вибирається невдало (наприклад, масив уже відсортований, зворотно відсортований або має V-подібну форму), алгоритм не може ефективно розділити дані. Складність деградує від ідеальних $O(N \log N)$ до найгірших $O(N^2)$, що викликає гігантські сплески часу. Те ж саме стосується масивів з винятково однаковими елементами через велику кількість зайвих обмінів.Практична поведінка в Linux (glibc): Системна функція qsort є високооптимізованою (часто використовує Merge Sort під капотом при достатній пам'яті). Тому вона вміє уникати $O(N^2)$ для "поганих" масивів. На практиці найзначніший час (сплеск) найчастіше фіксується на масивах із випадковими числами або майже відсортованих (з 1% хаосу) масивах, оскільки процесору набагато важче застосовувати передбачення розгалужень (branch prediction) при роботі з хаотичними даними порівняно з лінійними патернами.
